Conclusiones


* SerenityBDD con Screenplay permite la realización de las pruebas automatizadas de forma ordenada, clara y facilitando la mantenibilidad del código tanto para pruebas UI como para APIs. Adicionalmente, con la metodología BDD se desarrollaron los test a partir del comportamiento y esto reduce los errores en etapas más adelantes.

* Para este tipo de proyecto se puede usar tanto maven o glade para el manejo de dependencias en base a las necesidad del proyecto, en este caso fue maven ya que es un proyecto simple que no requiere personalizaciones complejas.

* Se crearon tres archivos poms, los cuales permiten tener las configuraciones globales y las necesarias para pruebas automatizadas UI y API de manera independiente y ordenada.

* Se usaron esperas explícitas ya que son solo algunos elementos los que lo requiere y se evita utilizar esperas como Thread.sleep lo cual podría afectar el rendimiento de la automatización.

* Se usó el manejo manual de WebDriver ya que era un proyecto pequeño pero cuando el proyecto es grande la configuración se puede hacer a través del archivo serenity.conf.

* Se parametrizan los datos en el task ya que permite la mantenibilidad de estos y no se encuentran hardcodeados pero para proyectos más grandes se puede hacer a través de un archivo externo.

* Otra de las ventajas de la estructura y frameworks usados en el proyecto es la generación de reportes automáticos, los cuales permitirían a través de su datos tomar decisiones, identificar cuellos de botellas, flujos con errores y evidencia de las ejecuciones.

* Se logró validar el comportamiento de los servicios REST de la API mediante operaciones CRUD, asegurando que cada endpoint responde correctamente según lo esperado (códigos HTTP y  contenido de la respuesta).

* No había usado karate antes pero creo que facilita la implementación de pruebas API de manera rápida, clara y mantenible
